## Строки

### Reverse String

**Идея.** String immutable → in-place реверс невозможен. Берём копию символов (`toCharArray()`), разворачиваем её two pointers, собираем строку обратно (`new String(chars)`).

**Решение (two pointers):**
```java
public static String reverse(String s) {
    if (s == null || s.length() < 2) return s;
    char[] chars = s.toCharArray();
    int left = 0, right = chars.length - 1;
    while (left < right) {
        char tmp = chars[left];
        chars[left] = chars[right];
        chars[right] = tmp;
        left++;
        right--;
    }
    return new String(chars);
}
```

**Альтернативы:**
- `new StringBuilder(s).reverse().toString()` — one-liner; на собесе обычно просят «без StringBuilder»
- Цикл с конца + StringBuilder.append — ок, но это тот же O(n) с лишним объектом
- Рекурсия `reverse(s.substring(1)) + s.charAt(0)` — O(n²) из-за substring/конкатенации + риск StackOverflow, упоминать только как анти-пример

**Сложность:** время O(n), память O(n) — от копии массива в Java не уйти (immutability).

**Ловушки:**
- `result += ch` в цикле — O(n²): каждая конкатенация копирует всё накопленное
- `chars.toString()` → `[C@1b6d3586` (дефолтный Object.toString), нужно `new String(chars)`
- Guard clause: `null` проверять первым, иначе NPE на `s.length()`
- `while (left < right)`, не `<=` — средний символ нечётной строки менять с самим собой не нужно
- Unicode-нюанс (senior-level): эмодзи и редкие символы — это surrogate pairs (2 char), посимвольный реверс их ломает. Для интервью достаточно проговорить, решать не требуют.

**Мои ошибки (ревью 2026-06-11):**
- Забыл `s == null` в guard clause → NPE на `s.length()`. Null — всегда первая проверка.
- Завернул основной код в `else` — guard clause означает ранний выход и плоский код дальше
- `static void main` без `public` — работает на Java 25+, но на собесе писать классическую сигнатуру

**Повтор по памяти (2026-06-12):**
- Guard clause и общая структура — правильно, по памяти ✅
- Ошибка в свопе: присвоил `chars[right]` в индекс `left` вместо `chars[left]` — перепутал индексы и элементы. Компилируется (char молча расширяется до int), но массив не меняется. В свопе двигаются только `chars[...]`, индексы — только `++`/`--`.
- `temp` объявил как `int` вместо `char` — следствие той же путаницы
- `main` снова без `public` — второй день подряд, дожать до автоматизма
- Статус: решено по памяти с 1 логической ошибкой 🔄 → ещё один повтор в начале следующей сессии

### Palindrome

**Идея.** Two pointers с краёв: нашли расхождение → сразу false; дошли до середины → true. Реверс не нужен — сравниваем на месте, без копии-разворота.

**Решение (базовый):**
```java
public static boolean isPalindrome(String s) {
    if (s == null) return false;
    if (s.length() < 2) return true;
    char[] chars = s.toCharArray();
    int left = 0, right = chars.length - 1;
    while (left < right) {
        if (chars[left] != chars[right]) return false;
        left++;
        right--;
    }
    return true;
}
```

**Усложнение (игнор регистра и пунктуации):** нормализация перед проверкой:
```java
String formatted = s.toLowerCase().replaceAll("[^a-zа-яё0-9]", "");
// Unicode-вариант: "[^\\p{L}\\p{Nd}]" — любая буква + десятичная цифра
```
Дальше — тот же базовый алгоритм. Альтернатива без regex (follow-up на собесе): `Character.isLetterOrDigit()` прямо в цикле — пропускать «мусорные» символы движением указателей, O(1) доппамяти.

**Нормализация без regex (предпочтительнее — O(1) память, частый follow-up «сделай без replaceAll»):**
```java
static boolean isPalindromeNormalized(String s) {
    if (s == null) return false;
    if (s.length() < 2) return true;
    int left = 0, right = s.length() - 1;
    while (left < right) {
        while (left < right && !Character.isLetterOrDigit(s.charAt(left)))  left++;   // перешагнуть мусор слева
        while (left < right && !Character.isLetterOrDigit(s.charAt(right))) right--;  // перешагнуть мусор справа
        if (Character.toLowerCase(s.charAt(left)) != Character.toLowerCase(s.charAt(right))) return false;
        left++;
        right--;
    }
    return true;
}
```
Внутренние `while` НЕ возвращают false и НЕ бегут до конца — они **пропускают** незначимые символы, сдвигая указатель до ближайшей буквы/цифры. Условие `while` — «пока продолжать двигаться»: `!isLetterOrDigit` = «пока это мусор». False даёт только внешнее сравнение, когда оба указателя стоят на значимых символах. Кириллица проходит бесплатно — `isLetterOrDigit`/`toLowerCase` понимают Unicode (в regex-варианте русские буквы пришлось бы вписывать явно).

**Сложность:** время O(n), память O(n) из-за `toCharArray()`; вариант с `charAt(left)` вместо массива — O(1) память.

**Ловушки:**
- Пустая строка и 1 символ — **палиндромы** (true). Guard clause не копипастится с reverse — возвращаемое значение зависит от семантики задачи.
- `return true` ставится **после** цикла: true = «прошёл все сравнения». Внутри цикла он срабатывает на первой итерации.
- `boolean`, не `Boolean` — обёртка может быть null → NPE на автораспаковке у вызывающего.
- Несовпадение — исключительный путь (`if (... != ...) return false;`), движение указателей — основной, без `else`.
- Противоречивые edge cases: если `""` → false, но `",."` после нормализации → true — логика расползлась. Null → false, остальное решает общий алгоритм.

**Мои ошибки (2026-06-12):**
- `return true` внутри цикла → «missing return statement» + сравнение только краёв
- `""` → false после явного указания, что пустая строка — палиндром
- Guard скопировал с reverse не думая (`< 2` → false)
- `Boolean` вместо `boolean`
- Регулярку для нормализации не знал — написал плейсхолдер. Запомнить: `[^a-zа-яё0-9]`
- Расширенный вариант не доделал — убрал нормализацию вместо исправления
- Статус: базовый ✅ (со 2-й попытки), нормализация 🔄 → повтор завтра

**Мои ошибки (2026-06-14, нормализация без regex):**
- `right++` вместо `right--` в конце цикла — правый указатель уползал вправо, выход за границы / StringIndexOutOfBounds. two pointers: left растёт, right убывает, сходятся в центре.
- Недопонимания (разобраны): думал, что внутренние `while` бегут до конца строки и возвращают false — на деле они пропускают мусор и встают на первой значимой букве; зачем `!` в `isLetterOrDigit` — условие `while` это «пока продолжать» = «пока мусор».
- Статус: нормализация ✅ (вариант без regex прогнан на 9 кейсах вкл. кириллицу)

### Anagram

**Идея.** Две строки — анаграммы, если набор символов одинаковый. Guard на разную длину отсекает сразу. Два подхода: сортировка (просто) и подсчёт (оптимально).

**Решение 1 — сортировка, O(n log n):**
```java
static boolean isAnagramSort(String a, String b) {
    if (a == null || b == null) return false;
    if (a.length() != b.length()) return false;
    char[] aChars = a.toCharArray();
    char[] bChars = b.toCharArray();
    Arrays.sort(aChars);
    Arrays.sort(bChars);
    return Arrays.equals(aChars, bChars);
}
```

**Решение 2 — счётчик, O(n), «правильный» ответ на собесе:**
```java
static boolean isAnagramCount(String a, String b) {
    if (a == null || b == null) return false;
    if (a.length() != b.length()) return false;
    int[] counts = new int[26];               // new int[] уже в нулях
    for (char ch : a.toCharArray()) counts[ch - 'a']++;
    for (char ch : b.toCharArray()) counts[ch - 'a']--;
    for (int count : counts) {
        if (count != 0) return false;
    }
    return true;
}
```

**Идиома `ch - 'a'`:** char — это число (код символа); вычитание `'a'` маппит `a..z` → `0..25`. Знать наизусть. Нужна **только** ради компактного `int[26]`: сжать диапазон кодов `97..122` в индексы `0..25`. С Map не нужна вовсе (ключ — сам символ).

**Размер массива = допущение о диапазоне символов (проговаривать вслух!):**
- `int[26]` + `c - 'a'` → только `a..z`
- `int[128]` + `counts[c]` (без вычитания) → вся ASCII: латиница, **цифры** (`'3'`=51), знаки, пробел. `'g'` попадёт в ячейку 103.
- кириллица: коды `U+0410`+ (`'а'`=1040, `'я'`=1103) — в `int[128]` дадут **ArrayIndexOutOfBoundsException**. Ловушка: `'a'`(97) и `'а'`(1040) выглядят одинаково, но разные символы.
- юникод/кириллица → `Map<Character,Integer>` (память O(k)) или `int[65536]` (весь char, но расточительно — на собесе минус).

**Аналогия массив ↔ мапа (одинаковый алгоритм, разное хранилище):**

| | `int[128]` | `Map<Character,Integer>` |
|---|---|---|
| адрес | индекс `counts[c]` | ключ `counts.get(c)` |
| значение | счётчик (int) | счётчик (Integer) |
| авто-ноль | `new int[]` сразу в нулях | `getOrDefault(c, 0)` подставляет 0 |
| плюс | `counts[c]++` | `put(c, getOrDefault(c,0)+1)` |
| минус | `counts[c]--` | `put(c, getOrDefault(c,0)-1)` |
| финал | проход, все == 0 | проход по `.values()`, все == 0 |

`getOrDefault(c, 0)` только **читает** (вернёт счётчик или 0, если ключа нет); запись — `put` снаружи. Это способ мапы имитировать «массив уже в нулях».

**Map: минус через `put(...-1)`, НЕ через `remove`!** `remove(c)` сносит запись целиком, игнорируя количество → ложный true. Контрпример `"aab"`/`"abb"` (длины равны, не анаграммы): remove-вариант даёт `{}`→true (баг), put-минус даёт `{a=1,b=-1}`→false ✅. Финал — проверка `.values()`, не `isEmpty()` (ключи остаются со значением 0).

**Сложность:** сортировка — O(n log n) время / O(n) память; счётчик — O(n) время / O(1) память. Память O(1) **потому что массив фиксированного размера** (не зависит от длины входа), а не «потому что массив один» — растущий-с-входом один массив был бы O(n). Map-вариант: память O(k) по уникальным символам, worst case O(n).

**Кирпичи сложности (запомнить):** сортировка → n log n; один проход/подсчёт → n; вложенный цикл → n²; поиск в отсортированном → log n.

**Ловушки:**
- `|` вместо `||` в guard — побитовое ИЛИ не короткозамкнутое, правая часть вычисляется всегда → NPE на `a.isEmpty()` при `a == null`
- Сортировка списка внутри цикла добавления — O(n²·log n); действию место после цикла
- Raw types (`List` без `<Character>`) — потеря типобезопасности
- `if (cond) return true; else return false;` → `return cond;`

**Мои ошибки (2026-06-12):**
- `|` вместо `||`; сначала переусложнил через `List` + `sort(null)` на каждой итерации вместо `Arrays.sort`/`Arrays.equals`
- Сложность сортировки назвал O(n) — сравнительная сортировка не бывает быстрее O(n log n)
- Счётчик начал и бросил (закомментировал) — второй случай отложенной сложной половины
- Статус: sort ✅, count ⬜ не дописан → повтор завтра + паттерн закрепится на «первом неповторяющемся символе»

**Повтор (2026-06-14) — count закрыт:**
- Написал `int[128]`-вариант сам: финал сначала `counts.length == 0` (всегда false — длина массива 128, не содержимое) → починил на проход с `i != 0`. Паттерн подсчёта руками наконец написан.
- Map-вариант: минус сделал через `remove` → ложный true на `"aab"`/`"abb"`; разобрали, починил на `put(...-1)` + `.values()`.
- Затыки в понимании (разобраны): `c - 'a'` зачем (сжатие диапазона под массив); `getOrDefault` путал с записью (только читает); ключ vs значение в мапе (символ = «индекс/адрес», int = значение).
- Статус: **count ✅** (оба: `int[128]` и Map), sort ✅. Anagram закрыт целиком.
- Метаобучение: пишет код по подсказкам — оседает слабо; **разбор после своего кода заходит сильно лучше** → меньше писать за него, больше спрашивать/разбирать.

### Частота символов

**Идея.** Один проход по строке, `Map<Character, Integer>`: символа нет → положить 1, есть → +1. Базовый паттерн подсчёта — фундамент для anagram count, first non-repeating, подсчёта пар, группировок.

**Решение:**
```java
static Map<Character, Integer> charFrequency(String s) {
    Map<Character, Integer> map = new HashMap<>();
    if (s == null) return map;                      // пустая мапа, не null!
    for (char c : s.toCharArray()) {
        map.put(c, map.getOrDefault(c, 0) + 1);
    }
    return map;
}
```

**Идиома `getOrDefault(key, def)`:** только читает — вернёт значение по ключу или `def`, если ключа нет; в мапу ничего не кладёт (put нужен снаружи). Схлопывает if/else (containsKey → get+1 / put 1) в одну строку: «первая встреча» — частный случай «было 0, стало 1».

**Сложность:** время O(n) — один проход по строке, операции HashMap O(1). Память O(k), k = число **уникальных** символов: worst case O(n), при фиксированном алфавите фактически O(1). Сильная формулировка: O(min(n, алфавит)).

**Ловушки:**
- Скобка внутри get: `map.get((c) + 1)` — ищется ключ `c+1` (int). Компилируется, т.к. `Map.get(Object)` принимает что угодно → вернёт null → в мапе появится null-значение. Опечатку в ключе компилятор не ловит.
- Guard: возвращать **пустую мапу**, не null — вызывающий код работает без null-проверок (общее правило: пустые коллекции вместо null).
- Печать всей мапы: `System.out.println(map)` → `{a=1, b=2}`; обход пар — `for (Map.Entry<Character, Integer> e : map.entrySet())`.
- Время обосновывать проходом **по строке**, не «по мапе»; память — уникальными символами, не размером строки.
- HashMap внутренности (если спросят отдельно): capacity 16, ресайз ×2 при числе записей > capacity × load factor (16 × 0.75 = 12).

**Мои ошибки (2026-06-12):**
- `map.get((c) + 1)` — скобка не на месте, искал по ключу-числу
- Не написал guard, хотя выбор поведения для null был частью задачи
- Вывод: цикл с `map.get('b')` × size раз вместо `println(map)`
- Сложность по времени обосновал «проходом по мапе» (проход — по строке); память начал объяснять через capacity 16 вместо O(k)
- Статус: ✅ оба варианта (if/else и getOrDefault)

### First Non-Repeating Character

**Идея.** Два последовательных прохода: 1) построить частотную мапу (= charFrequency); 2) пройти **по строке** и вернуть первый символ с `map.get(c) == 1`.

**Решение:**
```java
static Character firstNonRepeatingCharacter(String s) {
    if (s == null) return null;
    Map<Character, Integer> map = charFrequency(s);
    for (char c : s.toCharArray()) {
        if (map.get(c) == 1) return c;
    }
    return null;   // все символы повторяются
}
```

**Вопрос-ловушка: почему второй проход по строке, а не по мапе?** HashMap не гарантирует порядок (итерация по бакетам, не по порядку вставки) — «первизну» определяет строка. Бонус-ответ: с LinkedHashMap (хранит порядок вставки) можно было бы идти по мапе.

**Сложность:** время O(n) — два последовательных прохода; O(2n) не пишут, константы отбрасываются (характер роста линейный). Последовательные проходы — O(n), вложенные — O(n²). Память O(k) по уникальным символам; вход в доппамять не считается.

**Возврат «не найдено»:** `Character` + null — просто и ок для собеса; чище — `Optional<Character>` (`Optional.of(c)` / `Optional.empty()`), проговорить как плюс. Не возвращать String-сообщение — смешивает данные и текст, вызывающий код сравнивает с магической строкой.

**Ловушки:**
- Guard в каждом публичном методе: хелпер с guard'ом не спасёт, если `s.toCharArray()` зовётся до него
- `map.get(c) == 1` — автораспаковка Integer→int; при отсутствии ключа был бы NPE (здесь безопасно — мапа из той же строки; проговорить вслух)
- «Нет неповторяющихся» = все символы **повторяются** (не «все уникальны»)

**Мои ошибки (2026-06-12):**
- Guard остался в хелпере, основной метод падал с NPE на null
- Сентинел-сообщение «all characters are unique» — инвертированный смысл + String вместо Character
- Мёртвая переменная `chars`
- Сложность назвал O(2n)/O(2n) — константы не пишут; память посчитал со входом
- Статус: ✅ (ловушку про порядок HashMap раскусил сам, LinkedHashMap вспомнил сам)

### Valid Brackets ✅

**Задача.** `boolean isValid(String s)` — строка из `()[]{}`; валидна, если каждая открывающая закрывается парной в правильном порядке.
```
"()" → true   "()[]{}" → true   "([{}])" → true
"(]" → false  "([)]" → false    "((((" → false   ")(" → false
```

**Идея.** Стек (LIFO): открывающая → push, закрывающая → pop и сравнить с верхушкой. В конце стек пуст → true. В Java: `Deque<Character> stack = new ArrayDeque<>()` — класс `Stack` устарел, на собесе минус.

**Решение:**
```java
static boolean isValidBrackets(String s) {
    if (s == null || s.length() % 2 != 0) return false;  // нечётная длина — сразу false

    Deque<Character> stack = new ArrayDeque<>();
    for (char c : s.toCharArray()) {
        if (c == '(') stack.push(')');        // трюк: пушим ОЖИДАЕМУЮ закрывающую
        else if (c == '[') stack.push(']');
        else if (c == '{') stack.push('}');
        else {
            if (stack.isEmpty()) return false;   // ")(" — закрывающая при пустом стеке
            if (stack.pop() != c) return false;  // "(]", "([)]" — не парная
        }
    }
    return stack.isEmpty();   // "((((" — непустой стек = незакрытый долг
}
```

**Три случая провала (поймать все):** 1) не парна верхушке — `"(]"`; 2) закрывающая при пустом стеке — `")("`; 3) конец строки, стек не пуст — `"(((("`.

**Разобранное непонимание (важно!):** «как достать открывающую из глубины стека, если LIFO?» — НИКАК И НЕ НУЖНО. Сравнение всегда только с верхушкой: если закрывающая не совпала — строка невалидна по самому правилу вложенности (матрёшки: внутреннюю закрываем раньше внешней). LIFO — точная модель правила «последняя открытая закрывается первой». Оставшиеся в стеке скобки — «долг», непогашенный долг в конце = false.

**Трасса `"([{}])"`:** `(`→push`)` | `[`→push`]` | `{`→push`}` | `}`→pop`}`✓ | `]`→pop`]`✓ | `)`→pop`)`✓ | конец, пуст → true.
**Трасса `"([)]"`:** `(`→push`)` | `[`→push`]` | `)`→pop вернул `]` ≠ `)` → false.

**Сложность:** время O(n); память O(n) worst case — `"((((("` кладёт всю строку в стек. Ёмкость ArrayDeque (16, удвоение) — константные множители, в Big-O не входят.

**Ловушки:**
- `return true` после цикла вместо `return stack.isEmpty()` — пропускает `"(((("`. Коварно: тест `"((("` (нечётный) ловится guard'ом по длине и маскирует баг → тесты должны бить в каждый механизм, а не «хоть как-то падать»
- `c != '(' || c != '['` — всегда true (одно значение не равно хотя бы одному из списка); «не равен ни одному» — это `&&` (де Морган)
- Не плодить особых случаев для «первого символа» — пустой стек покроет
- Каждую открывающую — push, они копятся; «идём дальше» без push — потеря скобки
- Вопрос про мусор в строке — интервьюеру (контракт), а не молчаливый regex

**Мои ошибки (2026-06-12):**
- `||` вместо `&&` в «не открывающая» + `'{'` дважды, `'['` потерял
- `return false` на закрывающие — не было ветки для них вообще
- `return true` вместо `stack.isEmpty()`; тест `"((("` случайно проходил через guard по длине
- Снова тянуло в особые случаи (первый символ — отдельно, «не-скобки» в guard с переменной цикла, которой там не существует)
- Зачтено: guard на чётность — сам; трасса `"((([]"` с «долгом» — построил сам и правильно
- Статус: ✅ все 7 тестов

## Массивы

### Two Sum

Дан `int[] nums` и `target`, вернуть **индексы** двух элементов с суммой `target` (ровно одно решение, элемент дважды нельзя).

**Brute force — O(n²) время / O(1) память:**
```java
static int[] twoSum(int[] nums, int target) {
    if (nums.length < 2) return null;
    for (int i = 0; i < nums.length; i++) {
        for (int j = i + 1; j < nums.length; j++) {   // j от i+1: не сам с собой, без повторов пар
            if (nums[i] + nums[j] == target) return new int[]{i, j};
        }
    }
    return null;
}
```

**HashMap — O(n) время / O(n) память (дефолт на собесе):**
```java
static int[] twoSumHashMap(int[] nums, int target) {
    if (nums.length < 2) return null;
    Map<Integer, Integer> map = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int tail = target - nums[i];
        if (map.containsKey(tail)) return new int[]{map.get(tail), i};  // ищем среди ПРЕДЫДУЩИХ
        map.put(nums[i], i);                                            // кладём ПОСЛЕ проверки
    }
    return null;
}
```

**Идея HashMap:** вопрос не «перебрать все пары», а «встречал ли раньше число `target - nums[i]`?». Хранилище пройденного `Map<значение, индекс>` → проверка за O(1). Размен памяти на скорость.

**Почему ключ = значение, value = индекс:** искать надо ПО ЧИСЛУ (`containsKey`/`get` за O(1) работают по ключу) → число в ключ; вернуть надо индекс → индекс в value.

**Почему `put` ПОСЛЕ проверки:** иначе на `[3,3], target=6` найдёшь сам себя → `{0,0}` (элемент дважды). Проверяя до вставки, ищешь только среди предыдущих.

**Ловушки:**
- `for (int x : nums)` НЕ годится: даёт значение, а нужен индекс (вернуть `{i,j}` и считать `j=i+1`). **Паттерн «вложенный перебор пар» требует индексного `for`.** Даже при возврате значений двойной for-each сломан: складывает элемент сам с собой (`[4],8 → {4,4}`) — без индекса не отсечь.
- `map.get(tail) != 0` — ДВА бага: 1) если ключа нет, `get` вернёт `null` → распаковка `null != 0` = **NPE**; 2) индекс `0` валиден, `!= 0` пропустит ответ на нулевом индексе. Только `containsKey`.
- Возвращать индексы, а не значения (частая путаница в условии).
- `tail` объявлять внутри цикла (узкая область видимости).

**Two pointers тут НЕ подходит:** работает только на отсортированном массиве и теряет исходные индексы (а их надо вернуть). brute force / HashMap.

**Мои ошибки (2026-06-14):**
- Первая версия: завёл `right = length-1` (имена из two pointers), не двигал его → сравнивал каждый элемент только с последним; `while` отрабатывал один раз. Смешал brute force и two pointers.
- В HashMap-версии снова `for-each` с `nums[i]` (i как значение И как индекс одновременно); `map.get(tail) != 0` вместо `containsKey`.
- Зачтено: O-сложности проговорил сам верно; структуру brute force словами описал до кода правильно.

### Second Maximum

Дан `int[]`, вернуть **второе по величине различное** значение (`[5,5,3]` → 3, дубликат максимума не считается).

**Решение — один проход, два накопителя, O(n) время / O(1) память:**
```java
static Integer secondMax(int[] nums) {
    if (nums.length < 2) return null;        // второго максимума не существует
    int max = nums[0];
    Integer secondMax = null;                // Integer ради null = "ответа нет"
    for (int i = 1; i < nums.length; i++) {  // nums[0] уже в max → старт с i=1
        if (nums[i] > max) {
            secondMax = max;                 // СНАЧАЛА спасти старый max
            max = nums[i];                   // только ПОТОМ перезаписать
        } else if (nums[i] < max && (secondMax == null || nums[i] > secondMax)) {
            secondMax = nums[i];
        }
    }
    return secondMax;
}
```

**Идея.** Второй максимум — свойство **всего** массива, его нельзя вычислить, глядя на пары соседей. Несём через проход два накопителя `max`/`secondMax`, на каждом элементе сравниваем с **накопленным**, а не с соседом. Три ветки: больше max → max съезжает в secondMax; между secondMax и max → новый secondMax; иначе пропуск.

**Почему `< max` строгое:** дубликат максимума (`nums[i] == max`) не должен попадать в secondMax. При `==` не срабатывает ни ветка `> max`, ни ветка `< max` → корректный пропуск. `<=` сломал бы `[5,5,3]` (вернул бы 5).

**Почему `secondMax == null ||` первым в `||`:** secondMax — `Integer`, в начале `null`. Сравнение `nums[i] > secondMax` распаковывает Integer → NPE при null. Короткое замыкание `||`: если ещё null — правую часть не вычисляем.

**Сложность:** время O(n) — один проход; память **O(1)** — две переменные независимо от размера входа. Разделять: число проходов → время; объём доп. структур → память (новый массив/коллекция размером с вход дал бы O(n) памяти, две переменные — нет).

**Ловушки:**
- Сравнение **соседей** (`nums[i]` vs `nums[i+1]`) вместо накопителей — неверный подход в корне: соседи не знают про остальной массив, `secondMax` затирается последней парой. Плюс `nums[i+1]` при `i < length` → ArrayIndexOutOfBounds.
- **Порядок в первой ветке:** `max = nums[i]` ДО `secondMax = max` теряет старый максимум (оба становятся равны новому). Сначала спасти, потом перезаписать.
- Два `if` вместо `if/else if` — держится только на строгом `<`, хрупко; на ревью читается как «они точно взаимоисключающие?». `else if` делает намерение явным.
- `int secondMax = ...; return null;` не скомпилируется — примитив не бывает null. Возврат «ответа нет» → `Integer` + null (или исключение), проговорить выбор.

**Мои ошибки (2026-06-14):**
- Сначала сравнивал соседей `nums[i]` vs `nums[i+1]` → out of bounds + неверный алгоритм (вернул бы 2 вместо 4 на `[3,1,5,2,4]`). Разобрали: второй максимум — свойство всего массива, нужны накопители.
- Порядок присваивания в первой ветке перепутан (`max = nums[i]` до `secondMax = max`) → оба равны новому max, старый потерян. Сам отметил «стрёмно с порядком».
- Память назвал O(n) («один проход»); поправлено: проход → время, переменные → O(1) память.
- Инициализацию `max`/`secondMax` сам не собрал — подсказал (`max = nums[0]`, secondMax = null, цикл с i=1).
- Зачтено: гард `length < 2 → null` сам; вывод про ненужность вложенного цикла сам.

### Remove Duplicates (неотсортированный)

Дан `int[]`, убрать дубликаты, вернуть **новый** массив уникальных. Порядок сохраняем.

**Идея.** Дедуп = «видел ли раньше?» за O(1) → Set. Счётчики (HashMap) НЕ нужны — количество повторений не важно, берём все различные значения. Порядок: `HashSet` не хранит → `LinkedHashSet` (хранит порядок вставки).

**Решение — LinkedHashSet, O(n) время / O(n) память:**
```java
static int[] removeDuplicates(int[] nums) {
    if (nums.length < 2) return nums;
    Set<Integer> set = new LinkedHashSet<>();      // дубли отсеются сами при add
    for (int x : nums) set.add(x);

    int[] dedup = new int[set.size()];
    int idx = 0;                                    // индекс ведём руками
    for (int x : set) dedup[idx++] = x;             // постфикс: пишем в idx, потом ++
    return dedup;
}
```

**Перекладывание Set → int[]:** for-each даёт только **значение**, позицию держишь сам отдельным счётчиком. `dedup[idx++] = x` — постфиксный `++` возвращает текущий idx (пишем туда), увеличивает после → заполнение с 0 по порядку. Префиксный `++idx` пропустил бы нулевую ячейку.

**Стрим-вариант (короче, «знаю Stream API»):**
```java
return set.stream().mapToInt(Integer::intValue).toArray();
```
`mapToInt` обязателен: в сете `Integer`, обычный `Stream<Integer>.toArray()` вернул бы `Integer[]`, а нужен примитивный `int[]`. `mapToInt` → `IntStream` (поток примитивов) → его `.toArray()` отдаёт `int[]`. `Integer::intValue` — распаковка каждого элемента.

**Форма любой стрим-цепочки:** `источник.промежуточные.терминальная`. источник: `collection.stream()`/`Arrays.stream(arr)`; промежуточные (возвращают стрим): `filter/map/mapToInt/sorted/distinct`; терминальная (закрывает): `toArray/collect/count/forEach`. Не зубрить строку — собирать из слотов.

**Отсортированный вариант (follow-up «убери лишнюю память»):** дубли лежат рядом → two pointers in-place, O(1) доппамяти, без Set. Решён ниже.

**Сложность:** время O(n) — один проход (НЕ O(1)! один проход по n = линейно). Память O(n) — Set до n элементов.

**Ловушки:**
- `dedup[i] = i` в `for (int i : set)` — `i` это ЗНАЧЕНИЕ, не индекс. `dedup[3]=3` → ArrayIndexOutOfBounds (та же путаница «значение как индекс», что в Two Sum). Нужен отдельный `idx`.
- `HashSet` при требовании сохранить порядок → результат переставлен (`[4,2,4,1,2]` → `[1,2,4]` вместо `[4,2,1]`). Порядок важен → `LinkedHashSet`.
- HashMap со счётчиками — оверкилл (считаешь то, что не используешь); и логика «ключи где значение==1» — это first non-repeating, выкинет дубли совсем (`[1,1,2,3,3]`→`[2]`).
- `Stream<Integer>.toArray()` без `mapToInt` → `Integer[]`, не `int[]`.

**Мои ошибки (2026-06-14):**
- План был через HashMap+счётчики, шаг «взять ключи где значение==1» → это first non-repeating, не dedup; разобрали: для дедупа количество не важно, нужен Set со всеми различными.
- `dedup[i]=i` (значение как индекс) → out of bounds; разобрали ручной индекс `idx++` (постфикс vs префикс).
- Время назвал O(1) («один проход») → один проход = O(n).
- Спросил сам про более изящную конвертацию Set→массив (вышли на стрим `mapToInt`) и поднял тему запоминания синтаксиса Stream — дал каркас «источник.промежуточные.терминальная».
- Статус: ✅ три варианта (HashSet, LinkedHashSet, Stream) скомпилированы и прогнаны; отсортированный two-pointers вариант — на очереди.

### Remove Duplicates (отсортированный, in-place) ✅

Дан **отсортированный** `int[]`, убрать дубликаты **на месте**, вернуть `k` — число уникальных. Первые `k` элементов массива должны быть уникальны (LeetCode 26). Хвост после `k` — не важен.

**Идея.** Массив отсортирован → дубликаты лежат **подряд**. Это снимает потребность в Set: «видел ли раньше?» = «равен ли предыдущему уникальному?». Two pointers (fast/slow в одном массиве): `slow` — индекс последнего записанного уникального, `fast` сканирует. Новое значение (`nums[fast] != nums[slow]`) → сдвинуть slow и записать. O(1) доппамяти — главный выигрыш против Set-варианта.

**Решение — two pointers in-place, O(n) время / O(1) память:**
```java
static int removeDuplicatesSortedArray(int[] nums) {
    if (nums.length < 2) return nums.length;     // 0 или 1 — уже «уникальны»

    int slow = 0;                                 // nums[0] всегда уникален
    for (int fast = 1; fast < nums.length; fast++) {
        if (nums[fast] != nums[slow]) {           // отсортирован → дубли подряд
            slow++;
            nums[slow] = nums[fast];              // перезаписываем дубль уникальным
        }
    }
    return slow + 1;                              // slow — индекс → +1 = количество
}
```

**Два пойнтера в ОДНОМ массиве (не путать с palindrome two pointers):** там два указателя шли навстречу с краёв; здесь оба идут слева направо, `fast` всегда впереди, `slow` отстаёт и держит «границу уникальной зоны». Это паттерн **slow/fast (read/write)**, а не «навстречу».

**Контракт = число, не массив.** Возврат `int k`, массив мутируется на месте — в этом весь смысл (O(1) память). Хвост `nums[k..]` остаётся «мусором» (старые значения), вызывающий смотрит только первые `k`. Если нужен именно срез — `Arrays.copyOf(nums, k)` снаружи.

**Почему slow=0 / fast=1:** первый элемент уникален по определению (сравнивать не с чем) → он уже «записан», slow стоит на нём, скан начинаем со второго.

**Сложность:** время O(n) — один проход fast'ом; память **O(1)** — два индекса, без Set/нового массива. Главный контраст с неотсортированным вариантом: сортировка входа разменивается на отказ от O(n) памяти.

**Когда применим:** **только** на отсортированном входе. На неотсортированном `nums[fast] != nums[slow]` пропустит дубли, лежащие не рядом (`[1,2,1]` → не дедуплицируется) → там Set / LinkedHashSet.

**Ловушки:**
- Применить к неотсортированному — тихо неверный результат (не падает, просто дубли остаются). Проговорить предусловие вслух.
- `++slow` до записи или путаница порядка: сначала `slow++`, **потом** `nums[slow] = nums[fast]` — пишем в новую свободную ячейку, не затираем последний уникальный.
- Вернуть `slow` вместо `slow + 1` — off-by-one (индекс vs количество).
- Ждать «новый массив» как в неотсортированном варианте — здесь контракт другой (in-place + длина).

**Статус:** ✅ скомпилирован и прогнан (`[1,1,2,2,3,3]` → k=3). Паттерн slow/fast read-write закрыт — переиспользуется в «move zeroes», «remove element».

### Missing Number ✅

Дан `int[] nums` длины `n` с `n` различными числами из диапазона `[0, n]` (из `n+1` возможных ровно одно отсутствует). Вернуть пропущенное. Массив **не**отсортирован.

**Предусловие (контракт — проговорить вслух, спросить интервьюера):** числа строго из `0..n`, без дубликатов. Вне контракта (`[5,6,7]`, отрицательные, дубли) поведение не определено — оба метода вернут мусор. Хочешь защиту — валидация + `IllegalArgumentException`.

**Способ 1 — сумма Гаусса, O(n) / O(1):**
```java
static Integer missingNumber(int[] nums) {
    if (nums == null) return null;
    int n = nums.length;
    int sum = 0;
    for (int x : nums) sum += x;
    return n * (n + 1) / 2 - sum;     // ожидаемая(0..n) − фактическая = пропущенное
}
```
Формула Гаусса `0+1+...+n = n*(n+1)/2` (складывание с краёв к центру: `n` пар по `n+1`). Пропуск = насколько фактическая сумма меньше эталонной. `/2` всегда целое: из `n` и `n+1` одно чётное.

**Способ 2 — XOR, O(n) / O(1) (предпочтительнее: нет переполнения):**
```java
static Integer missingNumberXor(int[] nums) {
    int xor = nums.length;                  // засеваем n: цикл даёт индексы 0..n-1, индекс n массив не содержит
    for (int i = 0; i < nums.length; i++) { // nums[i] → граница i < length (НЕ length-1)
        xor = xor ^ i ^ nums[i];            // в «мешок» кидаем индекс И значение
    }
    return xor;
}
```

**Идея XOR («мешок»):** три факта — `x^x=0` (парные аннигилируют), `x^0=x` (ноль нейтрален), порядок не важен (коммутативность → сортировка НЕ нужна). Кидаем в мешок два набора: эталонные индексы `0..n` и фактические значения. Каждое присутствующее число встречается дважды (раз как индекс, раз как значение) → сокращается. Пропущенное есть только среди индексов, пары нет → остаётся. XOR = «выживает встретившееся нечётное число раз».

**XOR — побитовая операция, не сравнение:** `3^5`: `011 ^ 101 = 110 = 6` (бит=1 где биты различны). `x^x=0` именно потому, что у одинаковых все биты совпадают.

**Гаусс vs XOR:** одинаковая сложность; Гаусс рискует переполнить `int` на больших `n` (`n*(n+1)`) — лечится `long` или XOR; XOR переполнению не подвержен в принципе. На собесе показать оба + проговорить переполнение.

**Почему классический `for`, а не for-each:** нужны И индексы (как эталон `0..n`), И значения `nums[i]` → индексный `for`. (В Find the Difference на символах эталон не нужен — там for-each ок.)

**Связь с char/строками:** `char` — целое, XOR-трюк переносится: Find the Difference (LeetCode 389) — лишний символ между строками через `xor ^= c`. Single Number (LC 136) — тот же паттерн. XOR НЕ для «равны ли строки» (это `equals`), а для «найти непарного».

**Отсортированный вариант (другая задача):** на отсортированном входе — binary search по `nums[i] != i`, O(log n). Не путать с этим — здесь вход неотсортирован.

**Ловушки:**
- `int xor = 0` вместо `nums.length` → теряется верхушка диапазона `n`.
- Граница `i < length - 1` (приехала по инерции из задач с `nums[i+1]`) → пропускает последний элемент. Правило: трогаешь `nums[i]` → `i < length`; трогаешь `nums[i+1]` → `i < length - 1`. Смотри на самый дальний индекс в теле.
- Гарды «есть 0 → нет 1» и т.п. — ложный путь: пропуск может быть любым, формула/XOR ловят его без частных случаев.
- Смешать `+` и `^` (`xor + i ^ ...`) — `+` приоритетнее `^`, выходит мусор. В XOR-решении только `^`.

**Мои ошибки (2026-06-15):**
- Гарды на длину 1 (`length<2 && nums[0]==0 → 1`) — частные случаи вместо общей формулы; к тому же падали на пустом массиве (`nums[0]` при length 0). Убрали, оставили только `null`.
- XOR: паровал соседей `i` и `i+1` (как в Second Max) + смешал `+` с `^` + старт с 0 вместо `nums.length`. Разобрали мешок: XOR не сравнивает пары соседей, а сваливает всё в кучу.
- Решил, что XOR требует **отсортированный** массив и «индексы соответствуют значениям» — нет: порядок не важен (коммутативность), индексы — лишь способ сгенерировать эталон `0..n`.
- Спутал «`i` доходит до `length-1`» (значение) с условием `i < length-1` → off-by-one, терял последний элемент. Условие для значений `0..length-1` — это `i < length`.
- Затык в арифметике `3^5=6`: думал, XOR «сравнивает и выбирает»; разобрали побитово.
- Тест `[0,1,2]` маскировал баг границы (`nums[i]==i` → `i^nums[i]=0`, пропуск итерации незаметен); `[5,6,7]` (вне контракта) дал `1` — повод разобрать предусловия. Урок: тест должен бить в механизм, дружелюбный вход прячет баг.
- Статус: ✅ оба метода (Гаусс + XOR); XOR-идея усвоена через «мешок».
