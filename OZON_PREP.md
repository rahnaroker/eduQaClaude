## B. Тест-кейсы на код (живой формат Ozon)

Не теория, а навык: дают функцию → выдать набор кейсов. Техника:
- **Классы эквивалентности** — разбить вход на группы, где поведение одинаково; по 1 представителю на класс.
- **Граничные значения** — min, min−1, min+1, max, max−1, max+1, 0, пусто.
- **Негативные кейсы** — null, пустой вход, неверный тип, переполнение.
- **Позитивные** — типовой корректный сценарий.
- Структура кейса: предусловие → входные данные → шаги → ожидаемый результат.
- Дрилл: брать каждую решённую алго-задачу и писать на неё тест-кейсы.

## C. API / HTTP

- Методы HTTP: GET/POST/PUT/PATCH/DELETE/HEAD/OPTIONS; **идемпотентность** и безопасность.
- **POST vs PUT vs PATCH** (частый вопрос Ozon): создать / заменить целиком / частично обновить.
- Структура HTTP-запроса: стартовая строка, заголовки, тело; структура ответа, статус-коды (2xx/3xx/4xx/5xx).
- REST-принципы, что такое идемпотентность на примерах.
- Инструменты: Postman/curl; как тестировать API (позитив/негатив, валидация схемы, коды ответов).
- ⚠️ **Пробел — слои кэширования/инфраструктуры (подтянуть отдельно):** браузерный кэш, CDN, reverse-proxy (nginx), где что кэшируется, заголовки кэша (`Cache-Control`, `ETag`, `304`), как это влияет на тестирование API. Тема всплыла на разборе GET (safe→кэш) — пользователь плавает, изучить подробнее.
