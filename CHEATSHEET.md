# Шпаргалка

Краткие выжимки по пройденным темам. Формат: определение → нюансы → ловушки.

## Java Core

### OOP

**Overloading vs Overriding**
- Overloading — тот же класс, разные сигнатуры, resolved в compile-time (static dispatch)
- Overriding — дочерний класс, та же сигнатура, resolved в runtime (dynamic dispatch через vtable)
- `@Override` не обязателен, но ловит ошибки компиляции
- Ловушка: overloading с `null` — компилятор выбирает наиболее специфичный тип

**Static dispatch vs Dynamic dispatch**
- Static dispatch: компилятор вписывает конкретный метод в байткод (`invokestatic`) — overloading, static методы
- Dynamic dispatch: JVM смотрит на реальный тип объекта в рантайме через vtable (`invokevirtual`) — overriding
- Static метод в дочернем классе — не overriding, а **method hiding**: вызов по типу переменной, не объекта

**Abstract class vs Interface**

| | Abstract class | Interface |
|---|---|---|
| Поля с состоянием | ✅ | ❌ (только `static final`) |
| Конструктор | ✅ | ❌ |
| Множественное наследование | ❌ | ✅ |
| `default` методы | ✅ | ✅ (Java 8+) |

- Abstract class — когда есть общее состояние и частичная реализация (пример: `BaseTest`)
- Interface — когда нужен чистый контракт без привязки к иерархии
- Ловушка: abstract class может не содержать ни одного abstract метода — это валидно

**equals / hashCode**
- **Что такое hashCode:** метод `Object`, возвращает `int` — числовой «отпечаток» объекта. Хеш-коллекции по нему быстро находят **бакет** за O(1), не перебирая всё (аналогия — номер секции в гардеробе). Связка: **hashCode выбирает бакет, equals сравнивает внутри бакета.**
- Контракт: `equals == true` → `hashCode` обязан совпадать (обратное необязательно — одинаковый hash при разных объектах = **коллизия**, норма). + стабильность: пока объект не изменился, hashCode тот же.
- **hashCode — это НЕ:** уникальный id (бывают коллизии, int ограничен); НЕ адрес в памяти; НЕ проверка равенства (равенство — `equals`).
- Без `hashCode` (переопределил только equals) → два «равных» объекта дают разный hash → разные бакеты → HashMap «теряет» ключ, `get()` вернёт `null`.
- **equals и hashCode переопределяют ВМЕСТЕ по одним и тем же полям.** На практике — `Objects.hash(f1, f2)` + `equals` по тем же `f1, f2`.
- HashMap: `hash = hashCode ^ (hashCode >>> 16)` → `index = hash & (capacity-1)` → equals в бакете
- Capacity 16, load factor 0.75, resize × 2; бакет → дерево при ≥ 8 элементах

**== vs equals**
- `==` для объектов — сравнение ссылок; для примитивов — значений
- `equals()` — сравнение содержимого (если переопределён)
- Integer cache: -128..127 → `Integer.valueOf()` возвращает один объект → `==` даёт `true`; вне диапазона — `false`
- `new Integer()` всегда новый объект, кэш обходится (deprecated с Java 9)
- Autoboxing + `==`: `Integer x = 5; x == 5` — Integer анбоксится, сравниваются примитивы → `true`

**Конструкторы, this / super**
- Конструктор: имя = класс, нет возвращаемого типа (даже void нельзя), вызывается при `new`. Не наследуется, но перегружается.
- **Default-конструктор:** генерируется компилятором, только если НЕ объявлен ни один свой. Объявил любой → no-arg больше нет (`new User()` → ошибка компиляции).
- **`this.field`** — поле текущего объекта (отличить от параметра). **`this(...)`** — вызов другого конструктора этого класса (chaining, убрать дублирование).
- **`super.member`** — поле/метод родителя. **`super(...)`** — конструктор родителя.
- `this(...)` и `super(...)` — только **первой строкой** → нельзя оба в одном конструкторе.
- **Неявный `super()`:** если не написал this/super — компилятор вставляет `super()` первой строкой. Если у родителя нет no-arg конструктора → потомок обязан явно звать `super(args)`, иначе ошибка компиляции.
- **Порядок при `new Child()`:** `super(...)` → инициализаторы полей + instance-блоки потомка → тело конструктора потомка. (Раньше, при загрузке класса — static-поля/блоки, один раз.)
- Ловушки: конструктор с `void` — это обычный метод (тихий баг); вызов overridable-метода из конструктора — объект не достроен, поля потомка ещё null/0.
- **Зачем `super` (кратко):** доступ к родителю. `super(...)` — собрать родительскую часть объекта при создании; `super.method()/field` — взять родительскую версию (дополнить, а не выкинуть).
- **Зачем `this` (кратко):** ссылка на текущий объект. `this.field` — поле объекта (когда параметр назван так же); `this(...)` — вызвать другой конструктор этого же класса (без дублирования).
- Неявный `super()` есть ВСЕГДА (если не написал this/super). Вопрос лишь, найдётся ли no-arg у родителя и что он делает. Нет конструкторов в родителе → компилятор сгенерит пустой no-arg; только параметризованный → ошибка компиляции.

**Инициализация: примитивы vs обёртки**
- Значение по умолчанию зависит не от типа, а от МЕСТА: поле класса или локальная переменная.
- **Поля класса** — занулены автоматически: числа → `0`/`0.0`, `char` → `'\u0000'`, `boolean` → `false`, любая ссылка (обёртки, String, объекты) → `null`.
- Главное отличие: примитив-поле `int` → `0`; обёртка-поле `Integer` → `null` (обёртка = ссылка на объект, не значение).
- **Локальные переменные** — НЕ занулены. Чтение до присваивания → ошибка компиляции (и для примитива, и для обёртки одинаково).
- **Ловушка unboxing null:** обёртка-поле = `null`; распаковка в примитив (`int c = count + 1` при `count==null`) → NPE в рантайме. Классический вопрос «почему Integer-поле кидает NPE, а int нет».
- `final`-поле без инициализатора не зануляется — обязан задать конструктор, иначе не компилируется. `char` по умолчанию `'\u0000'` (нулевой символ), не пробел.

**Примитивы vs обёртки (вопрос на собесе)**
- Примитив хранит **само значение**; обёртка — **ссылку** на объект в куче. Размещение (стек/куча) зависит от контекста: локальный примитив → стек, примитив-**поле** → куча (внутри объекта). «Примитивы всегда в стеке» — неточно.
- **Зачем обёртки (по силе аргумента):**
  1. **Дженерики/коллекции только с объектами** — `List<int>` невозможен, только `List<Integer>`. Главная причина.
  2. **null = «не задано»** vs примитив обязан иметь значение (`int age = 0` — ложный «0 лет», не «неизвестно»). NULL-аемые поля БД → обёртки (связь с JPA/Entity).
  3. Память: `int` 4 байта vs `Integer` ~16 байт + ссылка → примитивы в горячем коде/больших массивах.
  4. `==`: обёртки сравнивают ссылки (+ Integer cache −128..127); примитивы — значения.
- Ловушка: unboxing `null` → NPE (достал `null` из БД → присвоил в `int` → упало).

**String pool: литерал vs new String() (вопрос на собесе)**
- Литерал `"hello"` → в **String Pool** (область в куче, по одной копии каждого литерала). Второй такой же литерал → та же ссылка, новый объект не создаётся.
- `new String("hello")` → **всегда** новый объект в куче (вне пула). Плюс литерал-аргумент тоже в пуле → потенциально **2 объекта**.
```java
String a = "hello";
String b = "hello";
String c = new String("hello");
a == b        // true  — один объект в пуле
a == c        // false — c отдельный объект в куче
a.equals(c)   // true  — содержимое равно
c.intern() == a  // true — intern() вернул объект из пула
```
- `==` сравнивает ссылки (ловит пул/не-пул), `equals()` — содержимое. Прямой аналог ловушки Integer cache.
- **Зачем пул:** экономия памяти — иммутабельную строку можно безопасно шарить одной копией. Immutability и делает шаринг безопасным (нельзя «испортить» чужую строку).
- Ловушки: `a == b` для двух литералов → `true` (контринтуитивно); `new String()` ломает шаринг → анти-паттерн, пиши литерал; пул в куче (с Java 7, до — PermGen); конкатенация переменных в рантайме → новый объект НЕ в пуле.

**Дженерики (Generics)**
- Параметризация типом: `<T>` — плейсхолдер, подставляется при использовании. Type-safe + без кастов.
- **Зачем:** переносят ошибку типа из рантайма (`ClassCastException`) в компайл-тайм. До Java 5 коллекции хранили `Object` → каст + риск падения в рантайме.
- Класс: `class Box<T> { T get(); }`. Метод: `<T> T first(List<T> l)` (параметр типа перед возвращаемым типом).
- Имена: `T` Type, `E` Element, `K`/`V` Key/Value, `N` Number, `R` Result.
- **Wildcards:** `List<?>` неизвестный; `List<? extends Number>` — Number и потомки (читать); `List<? super Integer>` — Integer и предки (писать). **PECS**: Producer Extends, Consumer Super.
- **Type Erasure (главный вопрос):** дженерики только на компиляции, в байткоде стираются до `Object`/границы. В рантайме `List<String>` и `List<Integer>` — один тип `List` (`a.getClass()==b.getClass()` → true).
- Следствия erasure: нельзя `new T()`, `new T[]`, `T.class`, `obj instanceof List<String>`.
- **Инвариантность:** `List<String>` НЕ является `List<Object>` (хотя String is-a Object) — для гибкости нужны wildcards.
- Примитивы нельзя: только `List<Integer>`, не `List<int>` (связь — зачем обёртки).

**String: иммутабельность → пул (логическая цепочка)**
- **Литерал** — значение прямо в коде (`"hello"`). Компилятор кладёт строковые литералы в String Pool.
- **Immutable** = объект нельзя изменить после создания. Методы String (`toUpperCase`, `substring`, `+`) не меняют объект, а возвращают **новый**:
```java
String s = "hello";
s.toUpperCase();        // вернул "HELLO", но s == "hello" (объект не тронут)
s = s.toUpperCase();    // s теперь смотрит на НОВЫЙ объект; старый — мусор
```
- **Почему String immutable:** безопасный шаринг (→ пул), потокобезопасность, кэш hashCode (быстрый ключ Map), безопасность (пути/URL не подменить после проверки).
- **Immutability ДЕЛАЕТ возможным пул:** раз строку нельзя изменить — один объект безопасно шарить между всеми ссылками (никто не «испортит» чужую). Будь String mutable — пул был бы невозможен (поменял через `a` → сломал `b`).
- **StringBuilder — mutable** аналог: меняет себя на месте, новый объект не создаёт. Для конкатенации в **цикле** (`s += i` в цикле = N объектов-мусора). Одиночный `a + b` — оставляй String (компилятор оптимизирует). `StringBuffer` — потокобезопасный StringBuilder, но медленнее.

**static**
- `static` поля/методы/блоки — принадлежат классу, не экземпляру
- Порядок инициализации: static поля+блоки (при загрузке класса) → instance поля+блоки → конструктор
- Static nested class — нет доступа к instance-членам outer → нет утечки памяти
- Inner (нестатический) class — держит неявную ссылку на outer (`Outer.this`) → риск утечки памяти

**instanceof / ClassCastException**
- `instanceof` — проверка типа объекта, используется перед кастом
- `ClassCastException` — runtime, только объектные типы; `float→int` — не исключение, это narrowing
- Pattern matching (Java 16+): `if (obj instanceof String s)` — каст + объявление переменной в одну строку

**final**
- `final class` — нельзя наследоваться; `final method` — нельзя переопределить
- `final field` — нельзя переприсвоить ссылку, но состояние объекта менять можно
- Ловушка: `final List` — `add/remove` работают, `=` — нет
- Инициализировать до конца конструктора обязательно

**Исключения: иерархия + checked/unchecked**
- Иерархия: `Throwable` → `Error` (фатальное JVM, не ловим: OOM, StackOverflow) и `Exception`. Под `Exception`: `RuntimeException` = **unchecked**, всё остальное = **checked**.
- **Checked** (IOException, SQLException): компилятор заставляет обработать или объявить `throws`. Смысл — внешний ожидаемый сбой (файл/сеть/БД), вызывающий **может восстановиться**.
- **Unchecked** (NPE, IllegalArgumentException, ArrayIndexOutOfBounds, ClassCastException): не проверяются компилятором. Смысл — **баг в коде** (null, неверный аргумент, выход за границы).
- Правило различения: checked = «снаружи пошло не так при корректном коде»; unchecked = «программист накосячил».
- **`throw` vs `throws`:** `throw` — оператор, бросает сейчас (`throw new X()`); `throws` — в сигнатуре, объявляет «могу бросить» (`void f() throws IOException`).
- **try-catch-finally:** catch'и от частного к общему (общий выше → unreachable, не компилируется); multi-catch `catch (A | B e)`; `finally` выполняется ВСЕГДА.
- **Ловушки:** (1) NPE — **unchecked** (топ-вопрос); (2) `return` в `finally` глотает исключение и затирает return из try — антипаттерн; (3) пустой `catch {}` — глотание бага; (4) `catch (Throwable)` ловит и Error — почти всегда ошибка; (5) `finally` НЕ выполнится только при `System.exit()` / убитой JVM; (6) исключения дорогие (заполнение stack trace) — не для control flow; (7) свой класс: `extends RuntimeException` (unchecked) или `Exception` (checked) по тому, обязан ли вызывающий обработать.

**try-with-resources (Java 7+)**
- Авто-закрытие ресурса по выходу из блока (успех или исключение). Ресурс объявляется в `try(...)`.
- **Синтаксис:** ресурсы в КРУГЛЫХ скобках после `try`, тело — в фигурных. Ресурс, созданный внутри `{ }`, НЕ закрывается сам (это обычный try). Несколько ресурсов — через `;`.
- Условие: ресурс реализует `AutoCloseable` (метод `close()`). `Closeable` — наследник, `close()` бросает только `IOException`; `AutoCloseable` — `Exception`.
- Закрытие в **обратном** порядке объявления (зависимый → базовый).
- `close()` вызывается **всегда**, даже при исключении в теле — заменяет `try/finally` с null-проверками.
- **Suppressed exceptions:** если бросают и тело, и `close()` — наружу летит исключение **из тела**, из `close()` подавляется и цепляется через `addSuppressed()` (достать — `getSuppressed()`). В старом `try/finally` было наоборот: исключение из `finally` затирало исходное.
- Java 9+: в `try(...)` можно передать уже объявленную effectively final переменную.

```java
try (var conn = ds.getConnection();
     var stmt = conn.prepareStatement(sql)) {   // закроется stmt, потом conn
    return stmt.executeQuery();
}
```
- Ловушки: наружу летит исключение из тела (не из close); обратный порядок закрытия; ресурс фактически final.

**Closeable vs AutoCloseable**

| | `AutoCloseable` | `Closeable` |
|---|---|---|
| Пакет | `java.lang` | `java.io` |
| Появился | Java 7 | Java 5 |
| `close()` бросает | `Exception` | `IOException` |
| Идемпотентность | не требуется спекой | требуется (повтор — no-op) |
| Для чего | любой ресурс (JDBC, Lock, свои) | I/O (стримы, файлы) |

- Иерархия: `Closeable extends AutoCloseable` (легко перепутать направление). TWR принимает `AutoCloseable` → работает с обоими.
- Зачем ввели `AutoCloseable`: `Closeable` ограничен `IOException`, а не всякий ресурс — I/O (JDBC бросает `SQLException`) → нужен общий надтип с `throws Exception`.

### Collections

**Map**
- Map не наследует Collection
- `HashMap` — O(1), без порядка, 1 null-ключ
- `LinkedHashMap` — insertion order, O(1)
- `TreeMap` — сортировка по ключу, O(log n), null-ключи запрещены, нужен Comparable/Comparator
- `ConcurrentHashMap` — многопоточка (вместо устаревшего Hashtable)

**List**
- `ArrayList`: get(i) O(1), add в конец O(1) амортизированно, вставка в середину O(n); компактная память
- `LinkedList`: doubly-linked list; get(i) O(n); add в head/tail O(1); реализует Deque
- На практике ArrayList быстрее LinkedList даже при вставках — cache locality
- LinkedList как очередь/стек когда нужны операции с обоих концов

## Теория тестирования

**Тест-пирамида**
```
      [  E2E / UI       ]   — медленно, дорого, flaky, мало
    [  Integration        ]  — средний слой, БД, сервисы
  [  Unit                  ] — быстро, изолировано, много
```
- Антипаттерн "ice cream cone" — мало юнитов, много UI. Признак legacy-проекта.

**Test Doubles**

| Double  | Реальная логика | Проверяет вызовы | Возвращает данные |
|---------|----------------|-----------------|------------------|
| Dummy   | нет            | нет             | нет              |
| Stub    | нет            | нет             | да (жёстко)      |
| Mock    | нет            | **да**          | можно настроить  |
| Spy     | **да**         | **да**          | реальные         |
| Fake    | да (упрощённая)| нет             | реальные         |

- Stub — `when(repo.findById(1)).thenReturn(user)`; Mock — `verify(emailService).send(...)`
- Spy нужен редко — его появление часто сигнал нарушения SRP
- В Mockito mock умеет и stubbing и verification — отсюда путаница в терминах

**Bug lifecycle (жизненный цикл бага)**
- Скелет: `New → Assigned → In Progress → Fixed/Resolved → Retest → Closed`; не починен → **Reopened** → назад в работу.
- Статусные модели у команд разные — описывать общим скелетом. Code-review статусы (Review/Reviewed) — это workflow кода, не классика бага.
- **Особые исходы (не путать!):**
  - **Rejected** — дефекта НЕТ: works as designed («не баг, а фича») / дубликат / не воспроизводится / ошибка тестера.
  - **Deferred/Postponed** — дефект ЕСТЬ, чиним позже (осознанно, в след. релиз). НЕ «забили и тухнет» — это намеренное решение триажа. Здоровый Deferred имеет ЦЕЛЬ (конкретный релиз); больной = «отложим бессрочно» → кладбище багов, раздувает **defect backlog** → главная мишень zero bug policy (заставляет выбрать fix-now или won't-fix).
  - **Won't Fix** — дефект есть, не чиним вообще (осознанно; валидно для zero bug policy). ≠ Rejected (Rejected = невалидный баг).
  - **Duplicate** — уже заведён ранее.
- **Cannot Reproduce:** главная причина — разные условия. QA прикладывает максимум контекста: точные шаги + тестовые данные, окружение (версия билда/ОС/браузер/конфиг), состояние данных, доказательства (логи, скриншот, **видео**, трейс). Алгоритм: воспроизвести у себя → сверить окружение → приложить логи/видео → совместная сессия → если плавает (интермиттент) = сигнал race/тайминги, не закрывать молча. Закрывать только когда обе стороны не смогли при одинаковых условиях.

**Severity vs Priority (часто путают)**
- **Severity** — техническая тяжесть бага (краш/потеря данных ↔ косметика). Объективна, **фиксирована**, ставит **QA**.
- **Priority** — срочность починки с точки зрения бизнеса. Субъективна, **меняется** со временем (косметика за день до релиза → high), ставит **PO/менеджер** (QA предлагает).
- Независимые оси → 4 комбинации:
  - High sev + high pri: краш в оплате → чинить немедленно.
  - High sev + low pri: краш в редкой/админской фиче → в очередь.
  - Low sev + high pri: опечатка в названии компании на главной → срочно (репутация).
  - Low sev + low pri: опечатка в подвале FAQ → бэклог / won't fix (связь с zero bug policy).

**7 принципов тестирования (ISTQB)**
1. **Наличие, не отсутствие дефектов** — тест показывает, что баги ЕСТЬ; доказать, что их нет, нельзя.
2. **Исчерпывающее тестирование невозможно** — все комбинации не перебрать → техники тест-дизайна.
3. **Раннее тестирование (shift-left)** — чем раньше, тем дешевле (баг на требованиях ≪ баг в проде).
4. **Кластеризация дефектов** — 80% багов в 20% модулей (Парето); нашёл один → копай рядом.
5. **Pesticide paradox** — неизменные тесты перестают находить новые баги. Борьба: пересматривать/обновлять тесты, добавлять новые, варьировать данные.
6. **Тестирование зависит от контекста** — банк и игру тестируют по-разному.
7. **Заблуждение об отсутствии ошибок (absence-of-errors fallacy)** — 0 багов ≠ хороший продукт, если он не отвечает нуждам пользователя.
- **Не путать 1 и 7:** 1 — нельзя доказать отсутствие багов (эпистемология); 7 — отсутствие багов не делает продукт нужным (бизнес-ценность).
- FIRST — это про юнит-тесты, НЕ принцип ISTQB (частая ошибка приписать).

**Pesticide paradox vs Defect clustering (частая путаница на собесе)**
- **Clustering** — свойство *продукта*: где баги сейчас (кучкуются в немногих модулях, 80/20). Ответ на вопрос «куда целиться».
- **Pesticide** — свойство *набора тестов*: почему со временем перестаёт находить новое. Ответ на «почему нельзя целиться одним и тем же вечно».
- Дополняют друг друга: clustering → «копай в модуле X»; pesticide → «но не одними и теми же тестами».
- **Почему старые тесты слепнут (как отвечать):** тест находит дефект 1 раз; после фикса он — регрессионная сетка (подтверждает, что старое не сломалось), но новых багов не ловит, т.к. баги уходят туда, где тестов ещё нет — в (1) новый/изменённый код, (2) непокрытые границы и комбинации данных. Лечение: ревизовать набор — добавлять кейсы, варьировать данные, применять новые техники тест-дизайна.
- **Ловушки:** (1) пестицид ≠ «удалять старые тесты» — их оставляют как регрессию, к ним добавляют новые; (2) лечит не частота прогонов, а изменение *содержания* тестов; (3) автоматизированная регрессия в CI — классический пример пестицида: дёшево ловит регрессии, к новым дефектам слепа → нужны новые автотесты под новую фичу; (4) если определил пестицид как «баги собираются в одном месте» — это провал, описал clustering.

**Хороший автотест (FIRST) + flaky (вопрос Ozon)**
- **FIRST:** **F**ast (быстрый), **I**ndependent/Isolated (любой порядок, без общего состояния), **R**epeatable (детерминизм — один результат всегда), **S**elf-validating (сам даёт pass/fail через assert), **T**imely (вовремя).
- **Структура тела:** AAA — Arrange (подготовка) → Act (действие) → Assert (проверка).
- **Одна причина падения** на тест (single responsibility), НЕ «ровно один assert»: несколько ассертов на одно поведение — ок; три разных сценария в одном тесте — плохо (упал — непонятно что).
- **Flaky-тест** — недетерминированный: на тех же данных то падает, то проходит. Вред: подрывает доверие к сьюту → реальные баги игнорят («опять флака»).
- **Причины flaky:** (1) общее состояние/грязные данные между тестами → изоляция + setup/teardown; (2) тайминги/асинхронность (`Thread.sleep`, race) → явное ожидание (Awaitility); (3) внешние зависимости (сеть, `now()`, рандом) → моки/фикс времени; (4) зависимость от порядка тестов.
- Repeatable и Independent из FIRST — прямое лекарство от flaky.
- **Heisenbug** — частный случай: баг исчезает/меняется при попытке наблюдать (дебаггер/логи меняют тайминги) → сильный сигнал на **race condition**. flaky/intermittent — общий термин «иногда воспроизводится»; heisenbug — именно «ускользает от наблюдения». (Зоология: Bohrbug — стабильно воспроизводим; Mandelbug — хаотичная причина; Schrödinbug — падает, как только прочитали код.)

**Фикстура (вопрос Ozon)**
- **Фикстура = фиксированное известное состояние**, в котором стартует тест (baseline): данные в БД/Кафке/ФС + подготовленные объекты, моки, соединения, экземпляр SUT. Это *что* (состояние), а `@Before*`/`@After*` — *как* (механизм управления). Не сливать понятия.
- **`@BeforeEach`/`@AfterEach`** — перед/после КАЖДОГО теста → свежая фикстура → изоляция + детерминизм (лечит flaky). **`@BeforeAll`/`@AfterAll`** — ОДИН раз на класс (в JUnit 5 — `static`) → быстрее, но общее состояние → риск flaky.
- Правило: дорогое read-only (поднять контейнер БД) → `@BeforeAll`; всё, что тест МЕНЯЕТ → `@BeforeEach` (чистый старт). Очистка данных в `@After*` ради Repeatable.

**Техники тест-дизайна**
- **Классы эквивалентности** — достаточно одного представителя от каждого класса (валид/невалид)
- **Граничные значения** — для возраста 1–120: проверяем 0, 1, 2, 119, 120, 121
- **Таблица решений** — для сложной логики с несколькими условиями. Условия × действия, каждый столбец = тест-кейс
- **Pairwise** — большинство дефектов вызваны взаимодействием двух параметров. Инструменты: PICT, AllPairs
- **State transition** — для объектов с жизненным циклом (заказ: создан → оплачен → отправлен)
- **Error guessing** — интуитивное, на основе опыта

**Тест-кейсы на функцию (живой формат Ozon: «дают функцию — протестируй»)**
- Группировать по технике: позитив (типовой) / классы эквивалентности / граничные / негатив. Проговаривать вслух **технику**, не россыпь примеров — интервьюер хочет увидеть систему.
- **Тестировать РЕАЛЬНЫЙ контракт, не выдуманный.** Нет проверки длины в коде → нет теста на лимит. Хочешь лимит — сначала валидация в коде (`if (len>255) throw`), потом тест на границу. Уточнить контракт у интервьюера ДО написания.
- **Оракул считать НЕЗАВИСИМО.** Ожидаемое — заданная руками константа (`reverse("boy")` → `"yob"`), НЕ вычисленная той же функцией (циклический оракул — сломанная функция пройдёт) и НЕ рандомом (недетерминизм → flaky, упавший тест не воспроизвести).
- **Параметризация — фиксированными парами** вход→выход, не случайными данными.
- **Типы ловит компилятор, не рантайм.** Передать `int` в `f(String)` → ошибка КОМПИЛЯЦИИ, не `ClassCastException`. Такой тест в Java не написать (в отличие от Python).
- Не путать вход и длину: `null` → `null` и `""` → `""` — РАЗНЫЕ кейсы с разным ожиданием. Строки длины −1 не существует.
- Формат кейса: предусловие → вход → шаги → ожидаемый результат. Для юнит-функции компактнее всего таблица «вход → ожидание → техника».

**Root Cause Analysis (RCA)**
- Поиск **первопричины** дефекта, а не симптома. Цель — устранить корень, чтобы класс багов не воспроизводился, а не залатать проявление.
- Симптом vs причина: тест упал с NPE (симптом) ← «нет валидации ответа внешнего API» (причина). Фикс одного теста = симптом; фикс валидации = корень.
- **5 Why** — главный инструмент: цепочка «почему?» до системной причины. (Платёж упал → 500 → таймаут БД → пул исчерпан → утечка соединений → не было try-with-resources ← корень). Фикс на уровне корня, не «добавить ретрай на 500».
- **Исикава (fishbone)** — категории причин: люди/процессы/инструменты/окружение/данные. **Парето** — 80% дефектов из 20% причин.
- Роль QA: на ретро/постмортеме приносит факты (шаги, логи, окружение, частоту). Найдя корень → регрессионный тест на этот класс проблем.
- Ловушки: путать симптом и причину («тест зелёный» ≠ корень устранён); остановиться на 1–2 «почему»; RCA — blameless, ищем причину в системе, не виноватого.

**Zero Bug Policy**
- Не «нет багов», а «не даём багам накапливаться»: число открытых дефектов держим около нуля.
- По каждому найденному багу решение **сразу**: либо **fix now** (приоритет выше новой фичи), либо **won't fix / close** (осознанно живём с ним). Третьего — «висит в беклоге на потом» — нет.
- Зачем: большой бэклог багов = иллюзия приоритизации (500 багов никто не разберёт) + стоимость триажа/перепроверки + гниение планки качества.
- Чинить сразу дешевле (контекст свежий) — связь со «стоимость дефекта растёт со временем». Тесно с DoD (фича не готова, пока есть её баги) и shift-left.
- Роль QA: быстро классифицировать (severity/priority, воспроизводимость, влияние) + не давать занижать баги до won't-fix ради скорости.
- Ловушки: понять буквально как «0 багов в продукте» (нет — 0 необработанных, won't-fix валиден); перепутать с «чинить всё» (часть осознанно закрывается); требует культуры «баг важнее фичи», иначе вырождается в массовый won't-fix.
- **Defect backlog ≠ tech debt:** defect backlog — копятся известные **дефекты** (в т.ч. Deferred); tech debt — компромиссы в **коде/архитектуре** (продукт работает, но менять дорого; «кредит скоростью сейчас → проценты замедлением потом»). Пересекаются, но разные кучи. Zero bug policy лечит именно defect backlog.

**Impact Analysis (анализ влияния)**
- Оценка **до изменения**: что оно затронет (модули, фичи, интеграции, тесты) → зона риска и что перепроверить. Изменение редко локально.
- Отвечает на «поправили X — что тестировать?» → очерчивает **scope регрессии**. Регрессия — сам прогон; impact analysis — что в этот прогон включить.
- Как: трассировка зависимостей (call hierarchy, кто вызывает), анализ по слоям (схема БД → Entity/DTO/миграции/отчёты), traceability matrix (требование↔кейс), git diff/blame, интеграционные точки (API-контракты, Kafka-консьюмеры).
- Три измерения: внутри кода (зависимые классы) / функциональное (фичи, сценарии) / внешнее (другие сервисы, контракты, БД).
- **vs RCA:** RCA — назад, почему сломалось (после дефекта); impact analysis — вперёд, что сломается при изменении (до). Связь с shift-left, change management (обязателен в финтехе перед изменением в проде).
- Ловушки: путать с RCA; сводить только к коду (есть ещё функц./внешнее); забыть транзитивные зависимости (A→B→C, правка C бьёт по A); это не «прогнать всё», а ВЫБРАТЬ минимально достаточный набор.

**Критерии приемки (AC)**
- Кто: PO — бизнес-смысл, QA — граничные/негативные кейсы, dev — технические ограничения («три амиго»)
- Когда: до начала разработки, на Refinement
- AC vs DoD: DoD — общий для всех историй; AC — специфичны для конкретной истории
- При невыполнении: story не закрывается, не идёт в релиз

**Definition of Done (DoD)**
- Единый проверяемый чек-лист «готовности» для **любой** задачи: код + review + тесты зелёные + CI + покрытие не упало + нет открытых багов по задаче + задеплоено/принято QA.
- **DoD vs AC:** DoD — общий для всех историй, отвечает «качество сделано правильно?» (как); AC — специфичны для одной истории, отвечают «реализовано то, что просили?» (что).
- Связка: история готова, когда выполнены И её AC, И общий DoD.
- Зачем: убирает разнобой в понимании «готово», не даёт протащить недоделку («тесты потом») в релиз — встраивает качество в процесс.
- Ловушки: путать с AC (DoD = как, AC = что); пункты DoD должны быть проверяемыми («review пройден», не «код качественный»). Зеркало — **DoR** (Definition of Ready): можно ли брать задачу в работу.

**STLC (Software Testing Life Cycle)**
- 6 фаз, у каждой — вход/выход (артефакт):
  1. **Requirement analysis** — требования на тестируемость, ищем пробелы/противоречия, вопросы к PO. → RTM (traceability matrix).
  2. **Test planning** — scope, ресурсы/роли, риски, оценка, критерии входа/выхода, инструменты. → **Test Plan**.
  3. **Test design** — кейсы, чек-листы, тест-данные, техники тест-дизайна. → тест-кейсы.
  4. **Environment setup** — тестовый стенд (часто **параллельно** фазе 3). → готовое окружение + smoke готовности.
  5. **Test execution** — прогон, факт vs ожидание, заведение дефектов, ретест+регресс. → отчёт о выполнении, баги.
  6. **Test cycle closure** — оценка по критериям выхода, метрики (defect density, покрытие), **lessons learned / ретро**. → Test Summary Report.
- **STLC ⊂ SDLC, но не зажат в фазе «тестирование»:** фазы 1 и 3 STLC идут **параллельно** требованиям/разработке SDLC (это и есть shift-left). SDLC: Requirements → Design → Development → Testing → Deployment → Maintenance.
- Ловушки: постмортем/lessons learned — это фаза 6 (closure), НЕ execution; путать STLC и SDLC (testing vs development).

**Shift-left / shift-right**
- **Shift-left** = двигать активности тестирования к началу таймлайна (вовлечь QA с фазы требований, кейсы параллельно коду). Это ровно принцип ISTQB #3 «early testing». Зачем: стоимость дефекта растёт экспоненциально по фазам (баг в требованиях ≪ баг в проде, Боэм — десятки/сотни раз). Ловушка: shift-left ≠ «QA пишет код», а раннее вовлечение в качество.
- **Shift-right** = тестирование/наблюдение **в проде** (то, что нельзя поймать раньше: реальная нагрузка/данные/поведение), но безопасно. Инструменты:
  - **Observability** — понять внутреннее состояние по внешним сигналам; 3 столпа: logs/metrics/traces. **vs Monitoring:** мониторинг следит за заранее известными метриками («*что* сломалось»), observability позволяет задать новый незапланированный вопрос («*почему*»). Мониторинг ⊂ observability.
  - **Canary** — релиз на малый % трафика (5→25→100), сравнение метрик с основной версией, автооткат при росте ошибок. Цель — *безопасность релиза*.
  - **A/B** — разным группам разные варианты, сравнение по бизнес-метрике. Цель — *выбор лучшего варианта* (продуктовый, не поиск багов). vs Canary: оба делят трафик, но canary про тех. безопасность, A/B про бизнес.
  - **Feature flags** — `if(flag.enabled)`, вкл/выкл фичу **без передеплоя**: гасить недоделку, катить на 1%, мгновенный откат. Разделяют **deploy** (код на серверах) и **release** (фича доступна юзерам) — частый вопрос.

**Test Strategy vs Test Plan**

| | Test Strategy | Test Plan |
|---|---|---|
| Уровень | организация/продукт в целом | конкретный проект/релиз |
| Горизонт | долгоживущий, статичный | под цикл, меняется от релиза |
| Кто | QA-лид/архитектор | тест-лид на проекте |
| Что | подход: уровни/виды тестов, инструменты, стандарты, общие риски | scope, расписание, ресурсы, окружения, критерии входа/выхода, оценка |
| Отвечает | «как мы вообще тестируем» | «что/кто/когда/где в этом релизе» |

- Plan **опирается** на Strategy и конкретизирует под проект. Стратегия одна — планов много.
- Нюанс Agile: стратегию часто встраивают разделом в тест-план, отдельного документа нет («по ISTQB два документа, на практике сливают»).
- Ловушки: Test Plan ≠ Test Case (план — организация процесса, кейс — шаги проверки); Test Plan — артефакт фазы 2 STLC.

## Backend-тестирование

**Идемпотентность HTTP**

| Метод | Идемпотентен | Безопасный |
|-------|-------------|-----------|
| GET, HEAD, OPTIONS | да | да |
| PUT, DELETE | да | нет |
| POST | нет | нет |
| PATCH | нет (по спеку) | нет |

- DELETE: коды разные (1-й → 200/204, повторный → 404), но эффект одинаков — норма
- PATCH не идемпотентен: `{"score": "+10"}` каждый раз меняет состояние

**POST vs PUT vs PATCH (частый вопрос Ozon)**

| | POST | PUT | PATCH |
|---|---|---|---|
| Семантика | создать | заменить целиком | обновить частично |
| Идемпотентность | ❌ нет | ✅ да | ⚠️ не гарантирована (зависит от патча) |
| URI | коллекция `POST /users` | конкретный `PUT /users/1` | конкретный `PATCH /users/1` |
| Тело | поля нового объекта | **весь** объект (опущенное поле → затрётся) | **только** меняемые поля |
| ID ресурса | назначает сервер | задаёт клиент (в URI) | задаёт клиент (в URI) |

- **Идемпотентность ≠ safe.** Safe (GET/HEAD) — вообще не меняет состояние. Идемпотентный — N повторов оставляют сервер в том же состоянии, что и 1 вызов (менять состояние может; PUT идемпотентен, но меняет).
- **PATCH идемпотентен или нет — зависит от операции:** `{name:"Анна"}` (set) — да; `{op:"increment"}` (прибавить) — нет. PUT идемпотентен **всегда** по спеку, PATCH — не обязан.
- **PUT vs PATCH — главное практическое:** PUT шлёт полное представление, опущенное поле обнуляется; PATCH трогает только переданные поля.
- **PUT может создавать** (upsert по известному URI). Отличие от POST: POST → создание в коллекции, ID даёт сервер; PUT → по адресу, заданному клиентом.
- **Миф «тело POST зашифровано» — нет.** Шифрует только HTTPS/TLS (и тело POST, и query GET одинаково). Плюс POST реальный: тело не в URL → не светится в логах/истории/Referer. Причина — «не в URL», не «зашифровано».

**Структура HTTP-запроса и ответа (вопрос Ozon)**

Запрос — 3 части + пустая строка-разделитель:
```
POST /user?active=true HTTP/1.1   ← стартовая строка: метод + путь(с query) + версия
Host: test.ru                     ← заголовки (пары ключ:значение)
Content-Type: application/json
                                  ← пустая строка (разделитель)
{ "name": "Анна" }                ← тело (опц.; у GET обычно нет)
```
Ответ — то же, но первая строка статусная: `HTTP/1.1 200 OK` (версия + код + reason-phrase) → заголовки → пустая строка → тело.

- **Параметры — это НЕ заголовки.** path-параметр в пути (`/user/5`), query-параметр в URL после `?` (`/user?id=5&sort=name`), заголовки — отдельная секция. Не путать.
- `scheme://host` в стартовую строку не идёт: host → заголовок `Host`, схема → уровень TLS. В стартовой строке только путь+query.
- Метод + путь + версия — всё в одной стартовой строке.

**Статус-коды**

| Класс | Смысл | Знать |
|---|---|---|
| 1xx | информация | 100 Continue, 101 Switching Protocols (WebSocket) |
| 2xx | успех | 200 OK, 201 Created, 204 No Content (успех без тела — DELETE/PUT) |
| 3xx | редирект | 301 (навсегда), 302 (временно), 304 Not Modified (кэш актуален) |
| 4xx | ошибка клиента | 400 Bad Request, 401, 403, 404, 405 Method Not Allowed, 409 Conflict, 422 Unprocessable Entity, 429 Too Many Requests |
| 5xx | ошибка сервера | 500, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout |

- **400 vs 422:** 400 — сломан синтаксис/формат; 422 — формат ок, не прошла бизнес-валидация.
- QA-практика: 204 → проверить, что тела нет; 429 → rate limiting; 304 → кэш.

**GET — свойства (вопрос Ozon)**
- **Safe** (только чтение, не меняет состояние) + **идемпотентен** + **кэшируется**.
- Идемпотентность ≠ «байт-в-байт одинаковый ответ»: это «запрос не меняет состояние сервера». GET к меняющемуся ресурсу может вернуть разные данные — всё равно safe/идемпотентен.
- Данные — в URL (path/query). Тело формально не запрещено, но без семантики → прокси/серверы игнорируют, полагаться нельзя.
- Чувствительные данные в GET не шлют: URL оседает в логах, истории, `Referer`, кэшах (причина — «URL запоминается везде», НЕ «GET не шифрует»; по HTTPS URL тоже шифруется в транзите).
- **Read-heavy ручки → GET:** safe→кэш (CDN/proxy перехватывают повторы, разгружают бэкенд) + безопасный ретрай при сбое сети (POST ретраить опасно — дубликаты). POST по умолчанию не кэшируется. Злонамеренный флуд кэш не лечит — нужен rate limiting (429).

**POST для чтения (поиск/фильтр)** — когда GET не тянет: длинный URL (`414 URI Too Long`, лимит ~2048), сложные вложенные фильтры удобнее JSON-телом, чувствительные критерии не в URL. Цена: теряются идемпотентность и кэш → применять точечно (`POST /search`).

**Authentication vs Authorization**
- Authentication — кто ты? → 401 если не опознан (название 401 Unauthorized обманчиво — значит «не аутентифицирован»)
- Authorization — что тебе можно? → 403 если опознан, но нет доступа

**Content-Type vs Accept**
- `Content-Type` — формат тела запроса который я отправляю
- `Accept` — формат ответа который я хочу получить. Сервер не может вернуть — 406

**Negative testing — чеклист**
- Пустые/невалидные поля → 400/422
- Несуществующий ресурс → 404
- Без токена → 401, чужой ресурс → 403
- Неверный метод → 405, дубликат → 409
- Граничные значения, SQL-инъекция в поле

**REST Assured — структура**
```java
given()  // baseUri, headers, contentType, body, auth
.when()  // .get/post/put/delete("/endpoint")
.then()  // statusCode(), body("field", equalTo(...)), extract()
```

**Contract testing (Pact)**
Consumer пишет тест → генерирует pact-файл → Pact Broker → Producer верифицирует в CI.
Проблема: у каждого сервиса тесты зелёные, но интеграция сломана из-за изменения контракта.

**POJO / Entity / DTO / DAO**
```
JSON → [Jackson] → RequestDTO → [Service] → Entity → [Hibernate] → БД
БД   → [Hibernate] → Entity → [Service] → ResponseDTO → [Jackson] → JSON
```

| Термин | Зачем | Логика | Привязка |
|---|---|---|---|
| **POJO** | зонтичный термин: обычный класс без привязки к фреймворку (и Entity, и DTO — POJO) | — | нет |
| **Entity** | отображение строки таблицы БД | минимум | JPA/Hibernate `@Entity`, `@Id` |
| **DTO** | передача данных между слоями/по сети | нет | сериализация JSON |
| **DAO/Repository** | доступ к хранилищу (CRUD: БД/Kafka/ФС) | да, логика доступа | хранилище |

- **DTO ≠ DAO** (частая путаница): DTO — переносит данные; DAO — обращается к хранилищу.
- Зачем Entity и DTO раздельно: не отдавать наружу чувствительные поля (`password`), развязать API-контракт от схемы БД.
- Entity сама JSON не разбирает и в таблицу не пишет
- `LazyInitializationException`: конвертировать Entity в DTO внутри `@Transactional`
- JPA — спецификация; Hibernate — реализация JPA (как JDBC и PostgreSQL Driver)

**CRUD ↔ HTTP ↔ SQL (как отвечать)**

CRUD — 4 базовые операции над данными. На собесе сразу маппь на HTTP и SQL:

| CRUD | HTTP | SQL | Идемпотентность |
|---|---|---|---|
| **C**reate | POST | INSERT | ❌ нет |
| **R**ead | GET | SELECT | ✅ да (+safe) |
| **U**pdate | PUT (целиком) / PATCH (частично) | UPDATE | PUT ✅ / PATCH ⚠️ |
| **D**elete | DELETE | DELETE | ✅ да |

- Связка: «CRUD — это абстракция операций, HTTP-методы — её реализация в REST, SQL-команды — в БД».
- Нюанс на засыпку: **POST ≠ всегда Create** (поиск через `POST /search`), **PUT может создавать** (upsert по URI). Маппинг семантический, не жёсткий.
- DELETE идемпотентен по эффекту: 1-й раз → 200/204, повтор → 404, но состояние сервера одинаковое.

**Тестирование пагинации**
- Обязательно: базовый позитив, последняя страница, за пределами (`page=999` → `[]`), невалидные параметры → 400
- Ловушка: без явного `sort` порядок недетерминирован → элемент может попасть на две страницы
- `size=10000` — сервер должен ограничивать максимум, иначе DoS

## БД (SQL)

**JOIN'ы**

| JOIN | Результат |
|------|-----------|
| INNER JOIN | только совпадающие строки с обеих сторон |
| LEFT JOIN | все из левой + совпадения из правой (NULL если нет) |
| FULL OUTER JOIN | все из обеих таблиц, NULL там где нет совпадения |
| CROSS JOIN | каждая строка левой × каждую строку правой, N×M строк |

- `INNER` — обязательная связь; `LEFT` — правая опциональна
- `FULL OUTER` — аудит расхождений; `CROSS JOIN` случайно в коде — обычно баг

**GROUP BY / WHERE / HAVING**
- `WHERE` — фильтрует строки **до** группировки, агрегатные функции нельзя
- `HAVING` — фильтрует группы **после** группировки, агрегатные функции можно
- `HAVING` вместо `WHERE` — работает, но медленнее

```sql
SELECT department, COUNT(*) as cnt
FROM employees
WHERE salary > 50000       -- до группировки
GROUP BY department
HAVING COUNT(*) > 5        -- после группировки
```

**NULL**
- `NULL = NULL` → не `true`, а `UNKNOWN`. Операторы `=`, `!=`, `<>` с NULL → `UNKNOWN`

```sql
WHERE email = NULL      -- всегда 0 строк → нужно IS NULL
NOT IN (1, 2, NULL)     -- вернёт 0 строк
AVG([1, 2, NULL, NULL]) -- вернёт 1.5 (NULL игнорируется агрегатами)
COUNT(*)                -- считает строки включая NULL
COUNT(field)            -- только не-NULL
```
- `COALESCE(field, default)` — заменяет NULL на дефолт
- `NOT EXISTS` вместо `NOT IN` когда подзапрос может вернуть NULL

**Индексы**
- Структура: B-tree, хранит отсортированные значения + указатели на строки. Поиск O(log n).
- Главный минус: каждый INSERT/UPDATE/DELETE обновляет все индексы таблицы
- Кардинальность: индекс на `is_deleted` (2 значения) бесполезен — оптимизатор сделает full scan

**PRIMARY KEY / FOREIGN KEY**
- PK: автоматически создаёт индекс. UUID v4 — фрагментирует B-tree; UUID v7 — монотонный, без фрагментации
- FK: referential integrity. `ON DELETE CASCADE` — удаляет дочерние; `ON DELETE SET NULL` — обнуляет FK
- Ловушка: FK **не** создаёт индекс автоматически в PostgreSQL → JOIN по FK без индекса → seq scan

**Транзакции и ACID**

| Буква | Свойство | Суть |
|-------|----------|------|
| A | Atomicity | Всё или ничего |
| C | Consistency | После транзакции все constraints выполняются |
| I | Isolation | Параллельные транзакции не видят промежуточные состояния |
| D | Durability | После коммита данные не потеряются (WAL на диске) |

**Уровни изоляции:**

| Уровень | Dirty Read | Non-repeatable Read | Phantom Read |
|---------|-----------|---------------------|--------------|
| READ UNCOMMITTED | да | да | да |
| READ COMMITTED | нет | да | да |
| REPEATABLE READ | нет | нет | да |
| SERIALIZABLE | нет | нет | нет |

PostgreSQL по умолчанию — READ COMMITTED.

**EXPLAIN / EXPLAIN ANALYZE**
- `EXPLAIN` — план оптимизатора, запрос не выполняется
- `EXPLAIN ANALYZE` — выполняет реально + фактическое время

```sql
BEGIN;
EXPLAIN ANALYZE DELETE FROM orders WHERE status = 'old';
ROLLBACK;
```

- `Seq Scan` — full scan, индекс не используется → плохо на больших таблицах
- `Index Scan` — индекс используется → хорошо
- `Bitmap Index Scan` — промежуточный вариант, лучше Seq Scan

**Subquery**
```sql
-- коррелированный (медленно — выполняется для каждой строки)
SELECT name, (SELECT COUNT(*) FROM orders WHERE user_id = u.id) FROM users u;

-- в WHERE
SELECT * FROM users WHERE id IN (SELECT user_id FROM orders WHERE status = 'paid');

-- в FROM (derived table)
SELECT * FROM (SELECT user_id, COUNT(*) as cnt FROM orders GROUP BY user_id) t WHERE t.cnt > 3;
```
- `JOIN` обычно быстрее — оптимизатор лучше строит план
- `IN (subquery)` на больших данных → медленнее чем `EXISTS` или `JOIN`

**Сериализация**
- Объект → байты/строка (сериализация). Байты/строка → объект (десериализация).
- JSON (Jackson): `mapper.writeValueAsString(obj)` / `mapper.readValue(json, Cls.class)`
- Java Serializable: без `serialVersionUID` при изменении класса → `InvalidClassException`
- Kafka: продюсер сериализует → консьюмер десериализует. Несовпадение → corrupted данные
- SERIALIZABLE в SQL — совпадение названия, не про сериализацию данных

---

**UNION / UNION ALL**
- Оба объединяют результаты двух запросов с одинаковым набором и типами колонок
- `UNION` — убирает дубликаты (неявный DISTINCT). Дороже — нужна сортировка/хэширование.
- `UNION ALL` — оставляет все строки включая дубликаты. Быстрее — просто склеивает.
- Типичная ошибка: писать `UNION` везде по привычке, хотя дублей нет — лишняя работа для БД.

```sql
-- A: [1,2,3], B: [2,3,4]
UNION     → 1, 2, 3, 4
UNION ALL → 1, 2, 3, 2, 3, 4
```

**DDL / DML / DCL**

| | Расшифровка | Команды |
|---|---|---|
| DDL | Data **Definition** Language | `CREATE`, `ALTER`, `DROP`, `TRUNCATE` |
| DML | Data **Manipulation** Language | `SELECT`, `INSERT`, `UPDATE`, `DELETE` |
| DCL | Data **Control** Language | `GRANT`, `REVOKE` |

- DDL — структура БД (схема); DML — данные внутри таблиц; DCL — права доступа
- Ловушка: `TRUNCATE` — DDL, не DML. В MySQL не откатывается транзакцией; в PostgreSQL — откатывается.

**CTE (Common Table Expression)**
- Именованный подзапрос, объявляется через `WITH` перед основным запросом
- Функционально = subquery, но читабельнее — сложный запрос разбивается на именованные шаги
- Можно объявить несколько CTE через запятую

```sql
WITH active_users AS (
    SELECT user_id, COUNT(*) as cnt
    FROM orders
    GROUP BY user_id
    HAVING COUNT(*) > 3
)
SELECT u.name FROM users u
JOIN active_users a ON u.id = a.user_id;
```

- `WITH RECURSIVE` — для иерархических данных (дерево категорий, оргструктура)
- Оптимизатор в большинстве случаев строит тот же план что и для subquery

**Window Functions**
- Агрегатные вычисления над набором строк без GROUP BY — строки не схлопываются
- Синтаксис: `функция() OVER (PARTITION BY поле ORDER BY поле)`
- `PARTITION BY` — аналог GROUP BY без схлопывания. Без него окно = вся таблица.

```sql
ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC)  -- номер строки в группе
RANK()        -- ранг с пропусками (1,2,2,4)
DENSE_RANK()  -- ранг без пропусков (1,2,2,3)
LAG(salary)  OVER (ORDER BY hire_date)   -- значение предыдущей строки
LEAD(salary) OVER (ORDER BY hire_date)   -- следующей строки
```

**Классический вопрос:** топ-1 сотрудник по зарплате в каждом отделе:
```sql
WITH ranked AS (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) as rn
    FROM employees
)
SELECT * FROM ranked WHERE rn = 1;
```

## Git

**merge vs rebase**

```
Старт:
main:    A - B - C
feature: A - B - D - E
```

`git merge main → feature` — создаёт merge-commit M, история нелинейная, D и E остаются с теми же SHA:
```
feature: A - B - D - E - M
```

`git rebase main → feature` — пересоздаёт D и E поверх C, новые SHA, история линейная:
```
feature: A - B - C - D' - E'
```

- `merge` — публичные ветки (main, develop)
- `rebase` — локальные feature-ветки перед PR
- Ловушка: rebase переписывает SHA → нельзя делать на уже запушенных коммитах

---

**reset vs revert**

`git reset` — двигает HEAD назад, переписывает историю:
```bash
git reset --soft HEAD~1   # коммит отменён, изменения в staging
git reset --mixed HEAD~1  # коммит отменён, изменения в working tree (дефолт)
git reset --hard HEAD~1   # коммит отменён, изменения уничтожены
```

`git revert` — создаёт новый коммит, отменяющий изменения указанного. История не переписывается:
```bash
git revert abc123
```

| | reset | revert |
|---|---|---|
| История | переписывается | сохраняется |
| Новый коммит | нет | да |
| Безопасно для shared веток | нет | да |

---

**stash**

Временно откладывает незакоммиченные изменения, чтобы переключиться на другую ветку.

```bash
git stash                     # отложить
git stash push -m "описание"  # отложить с именем
git stash list                # stash@{0}, stash@{1}...
git stash pop                 # вернуть последний + удалить
git stash apply               # вернуть, но оставить в stash
git stash drop stash@{1}      # удалить конкретный
git stash -u                  # включить untracked файлы (по умолчанию не попадают)
```

---

**cherry-pick**

Берёт конкретный коммит из любой ветки и применяет на текущую. Новый SHA, оригинал остаётся на месте.

```bash
git cherry-pick abc123
git cherry-pick abc123 def456  # несколько
```

Когда: hotfix попал не в ту ветку, закоммитил на неправильной ветке, нужна одна фича из большой ветки.
Ловушка: злоупотребление → дублирование в истории → признак плохого ветвления.

---

**fetch vs pull**

`git fetch` — скачивает изменения с remote, рабочую ветку не трогает.
`git pull` = fetch + merge (или rebase).

```bash
git fetch origin           # origin/main обновлён, локальный main — нет
git diff main origin/main  # посмотреть что изменилось
git pull origin main       # подтянуть и слить
git pull --rebase          # слить без merge-commit, история линейная
```

---

**merge conflict**

Триггеры: одна строка изменена по-разному, файл изменён и удалён, оба добавили файл с одинаковым именем.

```
<<<<<<< HEAD
return "hello world";
=======
return "hi there";
>>>>>>> feature-branch
```

```bash
git merge --abort          # отменить весь merge
git add filename           # отметить конфликт как решённый
git merge --continue
```

На практике: IntelliJ/VS Code → "Annotate with Git Blame" / визуальный diff с Accept Ours / Accept Theirs.

**blame**

Показывает для каждой строки файла: кто, когда и каким коммитом написал.

```bash
git blame UserService.java
```

Ловушка: показывает автора текущей версии строки, не оригинального. Рефакторинг/форматирование — blame покажет того кто форматировал. Полная картина: `git log -p filename`.

**Branching Strategy**

**Feature Branch Workflow** — самая распространённая: каждая фича = отдельная ветка → PR → merge в main.

**Gitflow:**
```
main       — только релизы (теги v1.0, v2.0)
develop    — интеграционная ветка
feature/*  — от develop
release/*  — подготовка релиза
hotfix/*   — срочный фикс от main
```

**Trunk-Based Development** — все коммитят в main напрямую (или ветки max 1-2 дня). Фичи скрыты за feature flags. Требует зрелого CI/CD.

| | Feature Branch | Gitflow | Trunk-Based |
|---|---|---|---|
| Сложность | низкая | высокая | низкая |
| Подходит для | большинства | несколько версий в проде | зрелые CI/CD команды |

## Docker

**Container vs VM**
- VM: гипервизор → гостевое ядро → процессы. Стартует минуты, образ гигабайты.
- Container: процесс на ядре хоста, изолированный через:
  - **namespaces** — изоляция PID, сети, файловой системы, hostname
  - **cgroups** — ограничение CPU, RAM, I/O
- Стартует секунды, образ мегабайты. Изоляция слабее: взлом ядра хоста → все контейнеры.
- Ловушка: Docker на Windows/Mac — внутри Linux VM (Docker Desktop). На Linux — напрямую.

**Dockerfile — основные инструкции**
- `FROM` — базовый образ
- `WORKDIR` — рабочая директория внутри контейнера
- `COPY` — копирует файлы с хоста в образ
- `RUN` — выполняется при **сборке** образа (установка пакетов)
- `EXPOSE` — документирует порт (не открывает реально)
- `ENV` — переменная окружения
- `CMD` — команда при **запуске** контейнера, можно переопределить в `docker run`
- `ENTRYPOINT` — фиксированный исполняемый файл, нельзя переопределить

```dockerfile
FROM eclipse-temurin:17-jre
WORKDIR /app
COPY target/myapp.jar app.jar
EXPOSE 8080
CMD ["java", "-jar", "app.jar"]
```

- `RUN` vs `CMD`: RUN — сборка, CMD — запуск
- `COPY` vs `ADD`: ADD умеет `.tar` и URL, но используй COPY если не нужно это
- Каждый `RUN` = новый слой → объединять через `&&` чтобы не раздувать образ
- `ENTRYPOINT` + `CMD`: ENTRYPOINT фиксирует команду, CMD задаёт аргументы по умолчанию

**docker-compose**
- Оркестрация нескольких контейнеров через один `docker-compose.yml`
- `image` — готовый образ; `build` — собрать из Dockerfile
- `ports` — `хост:контейнер`; `environment` — переменные окружения; `volumes` — персистентность данных
- `depends_on` — порядок старта, но не готовность сервиса (ловушка!)
- Все сервисы в одной сети → обращаются друг к другу по имени сервиса (`db`, `app`)

```bash
docker-compose up -d     # поднять в фоне
docker-compose down      # остановить + удалить контейнеры
docker-compose logs app  # логи сервиса
docker-compose ps        # статус
```

Для QA: `up` перед сьютом → прогон → `down`. Воспроизводимое изолированное окружение в CI.

**Основные команды Docker**
```bash
docker run -d -p 8080:80 --name myapp nginx  # запуск: фон, порт хост:контейнер, имя
docker run --rm nginx                         # удалить контейнер после остановки
docker ps / docker ps -a                      # живые / все контейнеры
docker exec -it myapp bash                    # зайти внутрь контейнера
docker logs -f myapp                          # логи в реальном времени
docker stop myapp / docker rm myapp           # остановить / удалить
docker rm -f myapp                            # остановить + удалить сразу
docker build -t myapp:1.0 .                   # собрать образ из Dockerfile
docker images / docker rmi nginx              # список образов / удалить образ
```
- `docker stop` — SIGTERM (graceful); `docker kill` — SIGKILL (немедленно)
- Для QA: `logs -f` — дебаг сервиса; `exec -it` — проверить файлы/переменные внутри; `ps -a` — понять почему упал

**Volumes**
- Контейнер эфемерный: удалил → данные пропали. Volume — директория вне контейнера.
- Именованный (`pgdata:/var/lib/postgresql/data`) — Docker управляет хранилищем
- Bind mount (`/host/path:/container/path`) — конкретная папка с хоста

| | Именованный | Bind mount |
|---|---|---|
| Хранение | Docker (`/var/lib/docker/volumes/`) | путь на хосте |
| Когда | данные БД | конфиги, логи |

- Для QA: `docker-compose down -v` — удалить контейнеры + volumes → чистая БД перед прогоном

**Docker в CI/CD**
- Пайплайн: `build jar → docker build → docker-compose up → tests → docker-compose down -v`
- Готовность контейнера — не retry в тестах, а `healthcheck` + `depends_on: condition: service_healthy`
- Testcontainers — поднимает контейнеры из Java-кода, гарантирует готовность к первому тесту
- Параллелизация сьютов: `docker-compose -p suite1 up` / `-p suite2 up` — разные namespace, нет конфликтов портов

**Docker Registry / Docker Hub**
- Хранит **образы** (не Dockerfile, не docker-compose)
- Путь: `docker build` → `docker push` → Registry → `docker pull` → сервер
- Docker Hub — публичный. В компаниях: AWS ECR, GitLab Registry, JFrog Artifactory
- Теги: `1.0.0` — для прода; `latest` — опасно (непредсказуемо); `main-abc123` — лучшая практика в CI
- Ловушка: `latest` не обновляется автоматически — это просто тег, нужно явно пушить
